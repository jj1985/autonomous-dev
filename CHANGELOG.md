### Fixed
- **Cross-session pipeline.json mixing no longer causes false completions or stale state** (#594): `AgentTracker` now reads `CLAUDE_SESSION_ID` at initialization and, when set, filters candidate `pipeline.json` files to those whose stored `claude_session_id` field matches the current session. If no matching file exists a new one is created; the fallback to the latest file is retained for backward compatibility when the env var is absent. `unified_session_tracker.py` applies the same filter in both `SessionTracker` (docs/sessions/*.md file selection) and `check_pipeline_complete()` (pipeline completion check), preventing a completing session from reading another session's pipeline state. Session files now include a `claude_session_id` field in their JSON structure. 6 new unit tests in `tests/unit/lib/test_agent_tracker_session_isolation.py` and 5 new unit tests in `tests/unit/hooks/test_unified_session_tracker_isolation.py`.


- **Pipeline intent validator no longer produces false INCOMPLETE findings in worktree context** (#593): When `validate_pipeline_intent()` ran inside a git worktree, session activity logs written by `session_activity_logger.py` were often stored in the main repository's `.claude/logs/activity/` directory rather than the worktree's own directory. Because the validator only read the local log path, pipeline events from the current session appeared absent and triggered spurious INCOMPLETE findings. `path_utils.get_main_repo_activity_log_dir()` was added to resolve the parent repository's activity log directory when running inside a worktree. `parse_session_logs()` gains an `additional_log_paths` parameter to merge events from supplemental JSONL files; duplicates are removed by `(timestamp, tool, session_id)` key before sorting. `validate_pipeline_intent()` automatically calls `get_main_repo_activity_log_dir()` and passes the matching log file as `additional_log_paths`, merging both log sources transparently. 9 new unit tests in `tests/unit/lib/test_worktree_log_resolution.py`.

- **False-positive env-var export detection no longer blocks CI analyst from filing GitHub issues** (#590): `_detect_env_spoofing()` in `unified_pre_tool.py` previously matched protected variable names (e.g. `CLAUDE_AGENT_NAME`) anywhere in the command string, including inside quoted `--body` arguments. When the continuous-improvement-analyst used `gh issue create --body "...CLAUDE_AGENT_NAME appeared in the session log..."` to file a GitHub issue with evidence text, the hook incorrectly treated the quoted argument value as an attempted identity spoof and blocked the command. Added `_strip_quoted_segments()` helper (Issue #590) that removes single- and double-quoted segments from the command string before pattern matching; `_detect_env_spoofing()` now operates on the stripped string. Actual inline spoofing patterns (`CLAUDE_AGENT_NAME=value cmd`) remain fully blocked; only quoted string contents are exempt. 12 new unit tests in `tests/unit/hooks/test_agent_identity_hardening.py` cover the false-positive scenarios alongside continued blocking of genuine spoof attempts.
- **Stale `implement_pipeline_state.json` from a prior session no longer blocks Bash writes** (#592): When a Claude session ended without cleanup, the leftover `/tmp/implement_pipeline_state.json` would be picked up by the next session's `unified_pre_tool.py`, which mistook it for an active pipeline and denied all infrastructure Bash writes. Added `_is_stale_session()` helper that compares the `session_id` stored in the state file against the current session's `CLAUDE_SESSION_ID`; when they differ (and both are known), the stale file is removed and the hook falls through to normal decision logic. `implement.md` pipeline activation code now writes `session_id` into the state dict at creation time so subsequent hooks can detect staleness. Staleness checks are applied in all three `_is_pipeline_active()` call sites that read the state file. 14 new unit tests in `tests/unit/hooks/test_stale_pipeline_state.py` cover stale detection, same-session pass-through, missing/unknown session_id fallback, and file removal. `tests/unit/hooks/test_unified_pre_tool.py` updated with test isolation fixes.
- **Pipeline subagents no longer denied writes due to absent `CLAUDE_AGENT_NAME` in subprocess env** (#591): `unified_pre_tool.py` previously resolved agent identity exclusively from the `CLAUDE_AGENT_NAME` environment variable. When the hook fires inside a subagent process, Claude Code does not propagate `CLAUDE_AGENT_NAME` to the subprocess environment, causing authorized pipeline agents (implementer, reviewer, etc.) to fail the agent-auth layer and have their writes blocked. Added `_get_active_agent_name()` helper that reads `agent_type` from the hook stdin JSON payload as the primary identity source (Claude Code always populates this field inside subagents) and falls back to `CLAUDE_AGENT_NAME` for compatibility. `_agent_type` module-level variable is extracted from stdin in `main()` and consumed by `_get_active_agent_name()` at all four former `os.getenv("CLAUDE_AGENT_NAME")` call sites. 8 new unit tests in `tests/unit/hooks/test_unified_pre_tool.py` cover the stdin-primary, env-fallback, and absent-identity paths.
- **Regression tests lock in Issue #572 Write-to-/tmp + cp bypass** (#572): Issue #572 identified that the Write tool correctly denied a direct write to `lib/reviewer_benchmark.py`, but the guard could be circumvented by writing staged content to `/tmp` first and then running `cp /tmp/staged_payload.py lib/reviewer_benchmark.py`. The cp/mv destination detection introduced in #589 covers this exact pattern. Five targeted regression tests in `tests/regression/test_bash_write_bypass_regression.py` (`TestIssue572WriteToTmpThenCp`) lock in this bypass route: exact cp command is extracted by `_extract_bash_file_writes`, blocked end-to-end by `_check_bash_infra_writes`, classified as a code-file target by `_is_code_file_target`; mv variant is also blocked; writes to `/tmp` itself remain allowed (precision check). These tests prevent silent regression of the fix that #589 delivered for this specific attack vector.
- **Bash write bypass via `Path.write_text()` and `sed -i` now fully blocked** (#589): `_extract_bash_file_writes()` previously only detected shell redirects (`>`, `>>`), `tee`, `cat > file << EOF`, and `dd of=FILE`. Wrapping a write in `python3 -c "Path('hooks/foo.py').write_text(...)"`, `sed -i`, `cp`/`mv`, or a python3 heredoc bypassed the code-file enforcement check in `_is_code_file_in_bash()`, even though `_check_bash_infra_writes()` caught these for infrastructure paths. Extended `_extract_bash_file_writes()` with four new detection patterns (sed -i in-place edits; cp/mv destination paths; python3 -c with `open()` write mode or `Path.write_text/write_bytes`; python3 heredoc bodies with the same patterns) so all Bash-wrapped writes are now caught by both enforcement paths. 12 new unit tests in `tests/unit/hooks/test_bash_write_bypass.py` and 17 regression tests in `tests/regression/test_bash_write_bypass_regression.py` (expanded from 2 by #572).
