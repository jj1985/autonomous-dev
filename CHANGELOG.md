### Fixed
- **File-based fallback for PIPELINE_ISSUE_NUMBER in hook invocations** (Issue #779): `_get_current_issue_number()` helper added to both `unified_pre_tool.py` and `unified_session_tracker.py`. Env vars set via `export` inside a Bash tool call do not persist to subsequent hook invocations because each Bash call runs in a fresh shell and hooks inherit env from the Claude Code parent process, not from prior Bash sessions. As a result, the ordering gate and prompt integrity gate were reading `PIPELINE_ISSUE_NUMBER` as `"0"` and recording/checking agent completions under the wrong issue number in batch mode, causing false-positive ordering blocks between batch issues. The helper resolves the issue number in two steps: (1) `PIPELINE_ISSUE_NUMBER` env var when it is set and non-zero, (2) the `issue_number` field in `/tmp/implement_pipeline_state.json` written by the coordinator, (3) safe default of `0`. `docs/BATCH-PROCESSING.md` updated to document the two-step resolution. 12 regression tests in `tests/regression/test_issue_779_env_var_fallback.py`.

- **Per-issue prompt baseline isolation in batch mode** (Issue #764): `get_prompt_baseline()` in `prompt_integrity.py` now accepts an optional `issue_number` keyword argument. When provided, it returns the baseline for that specific issue only, preventing a longer prompt from one issue seeding the baseline that then blocks a legitimately shorter prompt in a later issue (cross-issue false-positive shrinkage block). `unified_pre_tool.py` reads `PIPELINE_ISSUE_NUMBER` env var and passes it to `get_prompt_baseline()` so each batch issue compares only against its own recorded baseline. - **Block Python -c settings file write bypass in unified_pre_tool.py** (Issue #768): `_detect_settings_json_write()` previously only checked shell redirects, `tee`, `sed -i`, and `cp`/`mv` patterns.
